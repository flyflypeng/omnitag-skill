# 技能标题

## 概述

本技能专门用于根据用户提供内容信息结合用户标签系统管理规则，生成一组最佳的标签推荐给用户。

本技能参考了《打造第二大脑》中提到的PARA资源分类存储管理法则，然后结合多维度标签来共同描述内容的所属分类、内容主题、笔记类型、状态、以及元标签，帮助用户构建一套规则统一、层级清晰、结构稳定、易于过滤的标签系统。

现在用户在使用标签系遇到的几大问题痛点如下：

- **层级混乱**：`AI`​、`AI/Agent`​、`LLM` 等标签存在交叉重叠
- **分类标准不一**：有的按技术领域分，有的按笔记类型分，有的按状态分
- **颗粒度差异大**：`book-note`​ 很粗，`AI/Agent/Context-Engineering` 很细

**核心能力：**

- 维护一个json格式的全局标签数据文件`omnitag-data.json`，保存用户所有的标签数据
- 通过xxx工具将用户收藏文章的URL链接转换为LLM模型更易理解的Markdown格式，去除其他信息噪音
- 按照用户标签系统管理规则，通过LLM模型对用户输入的内容信息生成一组标签
- 支持用户不满意标签生成结果，可以重新生成

## 个人标签系统构建规则

### 顶层一级标签设计

基于PARA法则和你的使用习惯，我建议采用  **"3+1" 结构**：

```
📁 1. PARA核心分类（信息存储位置）- 必选
📁 2. 内容主题（信息所属领域）- 必选
📁 3. 笔记类型（信息所属类型）- 必选
📁 4. 元标签（系统级标记）- 可选
```

#### 关键设计原则

1. **PARA标签唯一性**：每条笔记必须有且只有一个PARA类型的标签，确保信息有明确归属
2. **主题标签开放性**：`#Topic/*`可以多选，反映笔记的跨领域特性
3. **类型标签描述形态**：`#Type/*`描述笔记是什么，而非讲什么
4. **元标签描述基础元数据信息**：`#Meta/*` 描述笔记的重要性、每日重复性工作标识等
5. **避免过度细分**：一级标签控制在20个以内，二级按需扩展

#### 1. PARA核心分类（必选其一）

| 一级标签          | 用途           | 示例二级标签                                                                           |
| ----------------- | -------------- | -------------------------------------------------------------------------------------- |
| ​`#PARA/Project`  | 当前进行的项目 | ​`/面试准备-20260330`​, `/OmniTag-20260415`                                            |
| ​`#PARA/Area`     | 长期维护的领域 | ​`/Agent系统`​ , `/云原生`​,  `/操作系统`​ , `/技术洞察`​ , `/职业规划`​ , `/知识管理` |
| ​`#PARA/Resource` | 感兴趣的资源   | ​`/AI前沿`​,  `/编程语言`​ , `/效能工具`​ , `/生活成长`​,  `/脑洞大开`​ , `/户外活动`  |
| ​`#PARA/Archive`  | 已归档的内容   | ​`/2025完成`​,  `/已暂停`                                                              |
| ​`#Inbox`         | 待处理的收件箱 | （无需二级）                                                                           |

**使用规则**：每条笔记**==必须有且只有一个PARA分类标签==**。

#### 3. 笔记类型（必选，描述笔记形态）

| 一级标签          | 用途          | 示例                    |
| ----------------- | ------------- | ----------------------- |
| ​`#Type/Note`     | 原始笔记/摘录 | flomo快速记录、会议记录 |
| ​`#Type/Insight`  | 洞察/思考     | 技术分析、观点输出      |
| ​`#Type/Paper`    | 论文相关      | 论文阅读、论文分析      |
| ​`#Type/Book`     | 读书笔记      | ​`book-note`迁移至此    |
| ​`#Type/Todo`     | 待办事项      | 需要后续行动            |

| 一级标签           | 用途                 |
| ------------------ | -------------------- |
| ​`#Meta/Daily`     | 每日记录             |

## 工作流程

### 1. 将用户输入信息转换为文本格式

- **核心原则**：将用户输入数据格式转换为LLM模型易用理解的文本格式
- 如果用户输入数据格式为纯文本格式，则无需做任务处理，传递给下一步处理即可
- 如果用户输入数据格式为URL网页连接，则可以调用xxx工具将URL网页转换为Markdown格式内容

### 2. 构建提示词

为了更好地知道LLM模型生成与内容最匹配的一组标签，需要在模型上下文窗口中必须包含以下内容：

- 用户全局标签数据文件`omnitag-data.json`​中的所有已有标签数据，默认存储路径：`~/.omnitag/omnitag-data.json`
- 用户标签系统构建规则和约束条件
- 用户输入的需要打标签的文本格式的内容数据信息
- 输出格式参考下面的形式：

  ```markdown

  推荐 Tag 列表：
  #PARA/tag1 #Topic/tag2 #Topic/tag3 #Type/tag4 #Meta/Daily

  推荐理由：
  ......
  ```

- 输出格式约束条件：

  - 如果输出的Tag的数量超过1个，则不同Tag之间用空格分隔
  - 推荐理由的中文字数不超过200字

### 3. 通过LLM模型生成推荐标签

- 将组装好的提示词发送给LLM模型，等待获取模型返回的结果，并将结果返回给用户
- 用户全局标签数据更新：将模型返回标签数据调用`xxx`​脚本，如果有新增的标签数据，则更新到全局标签数据文件`omnitag-data.json`文件中

## 示例

**场景1：你记录一条Agent论文笔记**

```
用户输入：
- 论文标题：A CPU-Centric Perspective on Agentic AI 
- 论文标注： "关键结论2：Agentic AI 工作负载的吞吐量要么被 CPU 因素（核心超额订阅、缓存一致性、同步）饱和，要么被 GPU 因素（显存容量、带宽）饱和"

输出内容：
- 推荐 Tag 列表：
#PARA/Area/Agent系统 #Topic/CPU #Topic/Workload  #Topic/Agent #Type/Paper
```
